## 1. はじめに

本カリキュラムでは、フロントエンド開発に必要な最新の技術スタックを使用して、実践的なポケモン図鑑アプリを作成します。以下の技術を使用します：

- **TypeScript**: JavaScriptのスーパーセットで、型安全性を提供します。
- **React**: ユーザーインターフェースを構築するためのライブラリ。
- **Vite**: 高速な開発環境を提供するビルドツール。
- **React Router**: クライアントサイドのルーティングを実現します。
- **Tailwind CSS**: ユーティリティファーストのCSSフレームワーク。
- **Tanstack Query**: データフェッチングとキャッシングのためのライブラリ。

アプリの機能として、ポケモンの一覧表示と詳細画面を日本語で提供します。また、レスポンシブデザインを採用し、モバイルファーストの実装を行います。

## 3. コマンドを打つ際の注意点

初学者がコマンドを打つ際に注意すべきなのは、そのコマンドを本当に実行していいかどうかわからない点です。
インターネット上の記事やチュートリアルで見つけたコマンドを深く理解しないまま実行すると、意図しないファイルの削除や、最悪の場合、セキュリティ上のリスクに繋がる可能性があります。

なので、それが本当に実行して問題ないかは、GitHub Copilot などに「このコマンドは何をするものですか？」と聞いてみることが非常に重要です。

それに加えて、以下の点にも注意しましょう。

### 現在の場所（カレントディレクトリ）を確認する

コマンドを実行する前に、まず自分が今どこにいるのかを確認する癖をつけましょう。ターミナルで `pwd` (print working directory) と打つと、現在のディレクトリパスが表示されます。

意図しない場所でコマンドを実行すると、「ファイルが見つからない」というエラーが出たり、逆に大切なファイルを上書きしてしまったりする可能性があります。特にファイルを作成・削除するコマンドの場合は、必ず `pwd` で場所を確認してから実行しましょう。

### `rm` コマンドは特に慎重に

`rm` はファイルを削除するコマンドです。特に `-rf` オプションを付けた `rm -rf` は、確認メッセージなしにディレクトリとその中身を強制的に全て削除するため、非常に強力で危険です。

例えば、プロジェクトの不要なファイルを消そうとして、誤って全く別の重要なディレクトリを消してしまう事故は後を絶ちません。`rm` コマンド、特に `rm -rf` を使う前は、削除対象のパスが本当に正しいか、指差し確認するくらいの慎重さが必要です。

### コピー＆ペーストに潜む罠

Webサイトからコマンドをコピーしてターミナルに貼り付ける際は、特に注意が必要です。悪意のあるサイトでは、見た目には普通のコマンドでも、実際には見えない改行や別のコマンドが仕込まれていることがあります。

安全のため、コピーしたコマンドは一度テキストエディタなどに貼り付けて、内容が完全にクリーンであることを確認してからターミナルで実行するようにしましょう。

### `sudo` は「伝家の宝刀」

`sudo` は、管理者権限（スーパーユーザー）でコマンドを実行するためのものです。システムの根幹に関わる設定を変更したり、重要なファイルを書き換えたりできるため、まさに「伝家の宝刀」です。

よくわからないまま `sudo` を付けてコマンドを実行すると、システム全体に影響を及ぼす可能性があります。エラーメッセージで安易に `sudo` の使用を勧められたとしても、そのコマンドが何をしているのか、なぜ管理者権限が必要なのかを理解してから使うようにしてください。

## 4. Reactをなぜ使うのか

Reactは、現代のWeb開発で最も人気のある「ライブラリ（道具箱）」の一つです。皆さんはこれまでに、生のHTMLとCSS、JavaScriptを使ってWebページを作成する経験を積んできました。その経験を踏まえ、まずはReactが登場する前の開発がどのような課題を抱えていたか、そしてReactがそれをどう解決するのかを見ていきましょう。

### 生のHTML開発の課題：手作業の限界

生のHTMLやJavaScriptでの開発は、小規模なWebサイトではうまく機能します。しかし、アプリケーションが複雑になるにつれて、いくつかの「辛い点」が顕著になります。

- **コードの重複との戦い**:
  多くのページで共通して使うヘッダーやフッターを想像してみてください。生のHTMLでは、その部分のコードを全てのHTMLファイルにコピー＆ペーストする必要があったはずです。もしヘッダーに新しいリンクを追加したくなったら、全てのファイルを一つ一つ手で修正しなければならず、非常に手間がかかり、修正漏れのリスクも高くなります。

- **複雑な状態管理**:
  ユーザーの操作によって画面の一部を書き換える（例えば、ボタンを押したら新しいデータを表示する）場合、JavaScriptを使って「どの要素を」「どのように変更するか」を細かく指示する必要がありました。`document.getElementById(...)` や `querySelector(...)` を使ってDOM要素を取得し、`textContent` や `innerHTML` を書き換える、という作業です。機能が増えるほどこの処理は複雑に絡み合い、コードは「スパゲッティコード」と呼ばれる、解読困難な状態に陥りがちです。

- **見通しの悪さ**:
  一つのHTMLファイルに、構造（HTML）、見た目（CSS）、そして動き（JavaScript）が混在し、さらにJavaScriptがDOMを直接操作することで、最終的に「この画面がどのように表示されているのか」をコードから直感的に理解することが難しくなります。

これらの課題は、例えるなら「**設計図なしに、手作業でレンガを一つ一つ積み上げて家を建てる**」ようなものです。Reactは、この家づくりを、もっと効率的で、見通しが良く、メンテナンスしやすいものに変えるための、画期的な「設計手法」と「道具」を提供してくれます。

### Reactを選ぶ理由：魔法のレシピという比喩

従来のWeb開発が「材料（HTML）と調理手順（JavaScript）を一つ一つ細かく指示する料理」だとすれば、Reactは「**完成形を伝えるだけで、あとはよしなに作ってくれる魔法のレシピ**」に喩えられます。

- **宣言的なUI**:
  レストランで「チーズバーガーが欲しい」と注文すれば、シェフが調理法を考えて提供してくれます。Reactも同様で、開発者は「画面がどのような見た目であるべきか」という完成形（状態）を宣言するだけでよいのです。細かい手順を指示せずとも、Reactが最も効率的な方法で画面を組み立て、更新を行います。これにより、コードはシンプルになり、間違いも起こりにくくなります。

- **コンポーネントベース**:
  React開発は、**LEGOブロック**で何かを組み立てる作業に似ています。UI（ユーザーインターフェース）を、「ボタン」「ヘッダー」「記事カード」といった、独立し再利用可能な部品（コンポーネント）に分割します。そして、これらのブロックを自由に組み合わせることで、一つの大きなWebページを構築していくのです。

- **豊富なエコシステムとコミュニティ**:
  Reactは世界中で利用されており、巨大なコミュニティが存在します。これは**「LEGOの達人たちが集まる巨大なクラブ」**のようなものです。問題が発生すればすぐに質問でき、「こんな部品が欲しい」と思えば、誰かが既に作成した便利な追加パーツ（ライブラリ）が多数見つかります。

### なぜコンポーネント化するのか？：LEGOブロックの利点

では、なぜUIを「コンポーネント」というLEGOブロックのように細かく分割するのか。それには、開発を劇的に効率化する多くの利点が存在するからです。

- **再利用性の向上**:
  一度「タイヤ」のブロックを作成すれば、車や飛行機など、他の制作物にもそのブロックを再利用できる。毎回ゼロから部品を作る必要がないため、開発速度が向上し、コードの重複も削減される。

- **保守性の向上**:
  もし「すべてのタイヤの色を赤に変更したい」場合、個々のタイヤを修正するのは非効率である。しかし、コンポーネントであれば、元の「タイヤ」ブロックの色を変更するだけで、それを使用するすべての乗り物のタイヤが一斉に更新される。これにより、修正と管理が非常に容易になる。

- **可読性の向上**:
  複雑なLEGOの城も、「門、塔、城壁」といったブロックの集合体として捉えれば、その構造は理解しやすくなる。コードも同様に、ページ全体が意味のある名前を持つコンポーネント（例：`PokemonCard`, `Header`, `SearchForm`）の組み合わせで表現されるため、誰が見ても直感的に構造を把握できる。

- **関心の分離**:
  各LEGOブロックは、それ自体で完結している。同様に、各コンポーネントも自身の見た目（HTMLのようなもの）とロジック（JavaScript）を内部に保持する。これにより、他の部分に影響されることなく、特定の部品の開発やテストに集中することが可能になる。

本カリキュラムで作成するポケモン図鑑アプリでも、これらの利点を最大限に活かすため、ページやUI部品を積極的にコンポーネントとして分割しながら開発を進めます。
